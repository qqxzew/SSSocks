Неделя 1: Инфраструктура, безопасность и CI/CD
День 1: Контейнеризация. Написание docker-compose.yml ровно на 4 контейнера: nginx, php-fpm, mysql, redis. Жестко закрываем порты: 3306 и 6379 доступны только внутри сети докера, наружу смотрит только веб-сервер.

День 2: CI/CD и проверки. Настройка GitHub Actions (push и pull_request). Пайплайн запускает: composer validate, линтер, PHPStan (уровень 8, без baseline) и PHPUnit.

День 3: Миграции и схема БД. Установка инструмента миграций. Написание структур для users, products, orders, order_items, idempotency_keys и reservations. Создание индексов на sku, user_id, order_id, idempotency_key. Определение ENUM статусов для заказа и резерва: pending, confirmed, cancelled, compensated, failed.

День 4: Безопасность ядра. Глобальное включение CSRF-токенов, настройка password_hash для авторизации. Настройка логирования Nette (Tracy) на запись ошибок чекаута и Lua-операций в выделенные файлы.

День 5: Защищенная админка. Написание middleware/презентера с проверкой is_admin. Реализация безопасной загрузки картинок для товаров: строгий MIME whitelist, лимит 2 MB, генерация UUID-имен. Файлы сохраняются вне web root (публичной директории), запрет на исполнение любых скриптов в папке загрузок.

День 6: Тестовое окружение. Поднятие тестовой БД в Docker. Написание фабрик для генерации фейковых товаров, пользователей и сессий для PHPUnit.

День 7: Резервный день.

Неделя 2: Highload-ядро, транзакции и отказоустойчивость
День 8: Каталог и безопасная корзина. Вывод витрины с кэшем. Корзина через Redis Hash (HINCRBY), чтобы параллельные запросы не затирали друг друга.

День 9: Идемпотентность и старт чекаута. Логика проверки idempotency_key. Если ключ уже есть в базе — возвращаем закэшированный ответ, блокируя повторную обработку.

День 10: Распределенная транзакция (Правильный флоу). Написание OrderService.

Создание записи в reservations со статусом pending (в MySQL).

Вызов атомарного Lua-скрипта в Redis (проверка остатка > 0 и списание).

Если Lua возвращает ошибку — смена статуса резерва на failed.

Если Lua успешен — старт транзакции MySQL (создание заказа).

При успехе транзакции — статус резерва confirmed.

При падении транзакции (сбой БД) — компенсация в Redis (INCR) и статус compensated.

День 11: Фоновый воркер (Cron). Написание CLI-команды Nette для очистки зависших резервов (на случай, если приложение упало между шагами 2 и 4). Скрипт ищет резервы pending старше 10–15 минут, возвращает остаток в Redis и помечает их как compensated/expired.

День 12: Интеграционные тесты (Стресс-тест). Тестирование OrderService. Проверка 50 успешных покупок из 100 одновременных запросов. Проверка отбрасывания дублей по idempotency_key. Симуляция падения MySQL после успешного Lua-резерва и проверка отсутствия двойной компенсации.

День 13: Скрытый терминал. Реализация консоли по тильде, генерация промокодов, Rate Limit на Lua-скриптах и TTL для кодов.

День 14: Резервный день. (Если чекаут потребует больше времени на отладку, терминал съезжает сюда или отменяется).

Неделя 3: Бизнес-фичи, Ops и документация
День 15: Апсейл в корзине. Внедрение "страховки от стиралки" с корректным пересчетом total_price до старта транзакции чекаута.

День 16: Геймификация. Tinder для носков. Если ядро работает нестабильно, этот день используется как резерв для багфиксов.

День 17: Динамические цены (Safe Mode). Сбор просмотров в Redis. Ограничение изменения цены (строгий коридор). Фиксация итоговой цены в order_items в момент покупки, сохранение логов изменения в price_history.

День 18: Личный кабинет. Базовый вывод статусов заказов для пользователя.

День 19: Ops и бэкапы. Настройка персистентности Redis (AOF + RDB). Настройка автоматического дампа MySQL. Проверка восстановления базы из бэкапа (чтобы убедиться, что дамп рабочий).

День 20: Документация (Часть 1). Оформление README. Обязательная вставка фразы: «MySQL — источник истины для заказов и постоянных данных; Redis используется для быстрых атомарных операций, кэша и временных резервов». Добавление блока про то, как система восстанавливается после падения (сценарий зависшего резерва и работа Cron-воркера).

День 21: Документация (Часть 2). Отрисовка схемы БД и Sequence Diagram полного процесса чекаута с отображением идемпотентности, Lua-скрипта, базы и механизма компенсаций. Подготовка к ревью.
